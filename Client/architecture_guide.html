Executive Summary & Core Metrics

This document presents the complete system architecture for the portfolio management system of EGA AJITH G. The platform is designed as a secure, full-stack application connecting a modern, high-performance React frontend via Vite with a secure Express.js API gateway and a robust MongoDB instance managed via Mongoose schemas.

The application facilitates complete control over public portfolio assets (projects, certifications, skills) while capturing telemetry, visitor traffic, and system errors directly into an administrative console. Unauthorized access is blocked using state-of-the-art encryption middleware alongside dual visual layouts tailored for administrator operations.

Theme & Sound State Synchronization Lifecycle

To deliver a cohesive interactive experience across separate layouts, the React client implements a localized synchronization pattern via localStorage hooks:

  • Theme Sync: A state hook in the root layout monitors the adminTheme key. When modified, the hook applies class lists like .cyber-red or .corporate-blue to the <body> element, enabling isolated component stylesheets to toggle color schemes instantly.
  • Audio Preferences Sync: User preference for audio feedback is persisted under the sound_fx_enabled key (boolean). The global mute/unmute command inside the CLI Terminal or controls within the UI write to this setting.
  • Audio Context Initialization: Due to modern browser autoplay security policies, the audio engine delays initializing the Web Audio AudioContext until the first user interaction (click or keypress), avoiding console exceptions.

Server Side

Backend Server Architecture & Data Flow

The server layer acts as a RESTful controller gateway. It connects routes directly to MongoDB model schema controllers and filters incoming queries using validation middleware.

Server Mechanics

  • CORS Intercepts: Restricts cross-origin resource sharing, permitting only verified client origin routes.
  • Payload Constraints: Configured body-parser limits set to 10MB to facilitate base64 image transfers from the portfolio admin dashboard.
  • System Error Ledger: An active global cache registry (systemErrorLedger) stores the latest 20 database or runtime error logs, feeding details directly to dashboard telemetry charts.

Security Middleware & JWT

All administrative database modifications require dynamic signature checks. Requests must attach a verified authentication token in the request header:

Authorization: Bearer <JSON_WEB_TOKEN>

The Express middleware parses the token with the server's private JWT_SECRET, rejecting malformed requests with 401 Denied status codes and structured JSON alerts.

System Data Flow & Request Lifecycle

Password Hashing & Hashing Mechanics (BCrypt)

To prevent credentials exposure in the database, the server never stores plain-text passwords. It utilizes bcryptjs to perform one-way cryptographic hashing:

  • Salt Generation: A cryptographically secure salt is generated using 10 rounds of computational complexity. This slows down brute-force and dictionary attacks.
  • Schema Pre-Save Hook: The Mongoose AdminSchema intercepts save and update operations. If the password field is modified, it generates a salt, hashes the input password, and overwrites the plain-text value: 
    AdminSchema.pre('save', async function(next) {
  if (!this.isModified('password')) return next();
  const salt = await bcrypt.genSalt(10);
  this.password = await bcrypt.hash(this.password, salt);
  next();
});
  • Signature Verification: When logging in, the server retrieves the document by username and uses bcrypt.compare() to safely match the input password with the stored hash, preventing timing attacks.

Data Sanitization & Injection Prevention

Security measures are implemented at the router and controller layers to shield the system from common web vulnerabilities:

  • NoSQL Injection Shielding: Query parameters from untrusted clients are cast explicitly to strings or verified object types. Mongoose strict schema casting prevents query operators (like $gt or $ne) from being injected into filters.
  • XSS (Cross-Site Scripting) Defense: Form inputs from the public Contact page (name, message, email) are sanitized. The React client renders user input as plain text nodes (automatically escaping HTML tags), and the backend validates inputs against strict string schemas before storage.

System Vulnerability & Bug Audit Ledger

A static audit analysis of client modules, controller scripts, and Express route routers identifies security vulnerabilities and operational defects. Remediation paths are detailed below:

🟢 1. No Rate Limiting on Contact Form Transmissions & Traffic Hits
POST /api/messages | POST /api/analytics/hit
Public endpoints were unprotected by rate limiters, posing database flooding risks.
Remediation Applied: Implemented a custom in-memory rate limiting middleware (rateLimiter.js) that registers and tracks IP hit sequences: 
// Limits messages to 5 submissions per 15 minutes
// Limits analytics hit registration to 3 requests per second
🟢 2. Unhandled Database CastError on Invalid ObjectIDs
DELETE /api/messages/:id | PUT /api/projects/:id | PUT /api/certifications/:id | PUT /api/skills/:id
Malformed ObjectIDs caused uncaught database exceptions and leaked system stack traces on 500 responses.
Remediation Applied: Created validateObjectId.js middleware validation intercepting param requests and responding with a standard error message: 
if (!mongoose.Types.ObjectId.isValid(req.params.id)) {
  return res.status(400).json({ error: 'INVALID_ID' });
}
🟢 3. Missing JWT Token Revocation (Session Logouts)
POST /api/auth/logout | Authorization Middleware
Tokens remained active and valid until expiration on logouts, risking admin access session hijackings.
Remediation Applied: Created a Mongoose-backed TokenBlacklist model caching revoked sessions upon logout with auto-expiring records via MongoDB TTL indexing (expireAfterSeconds: 0 on JWT expiry time).
🟢 4. Potential Base64 Large Payload Bloating in Database
POST/PUT /api/projects | POST/PUT /api/certifications
Direct unconstrained base64 uploads risked hitting MongoDB's 16MB document cap and bloating collection storage.
Remediation Applied: Added API payload size constraints in Mongoose route files rejecting image base64 strings exceeding 2MB and returning a clean 400 Payload Too Large payload response.
🟢 5. Lack of Password Complexity Policy on Change-Password
POST /api/auth/change-password
Weak administrative passphrases could be configured, leaving the console vulnerable to brute-force attempts.
Remediation Applied: Implemented input strength validation using an Express regex assertion matching strict requirements (minimum 8 characters, at least 1 uppercase letter, 1 lowercase letter, and 1 numeric digit) before triggering hashes.
🟢 6. Unlimited Growth of uniqueIps Array in Analytics Document (Database Bloat)
Models/Analytics.js | POST /api/analytics/hit
IP storage array in the telemetry statistics document grew indefinitely, risking 16MB document bounds crash.
Remediation Applied: Removed uniqueIps array. Created dedicated Visit model storing daily hits with a automatic 30-day Mongoose TTL deletion sweep index. Stats increments are calculated through visit presence.
🟢 7. Localized State Desynchronization for Mechanical Sounds Preferences
Client TerminalWidget.jsx | Client soundUtils.js
Sound state changes inside CLI terminal desynchronized from the rest of the application.
Remediation Applied: Exposed custom event dispatching (soundFXMuteToggled) in soundUtils.js and registered active state hook listeners in TerminalWidget.jsx to synchronize audio preferences instantly.
🟢 8. SPA Wildcard Rewrite Overrides Public API Routes in Vercel Production
Client vercel.json
Production Vercel wildcard rules captured public REST requests under /api/* and returned HTML instead.
Remediation Applied: Updated vercel.json rewrites mapping API subpaths directly to the production Railway backend API gateway (https://ega-ajith-portfolio-server.up.railway.app/api/:path*) before wildcard SPA rule matches.

Troubleshooting Diagnostics Ledger

Problem / Error Likely Cause Remediation Protocol
"Error connecting to MongoDB..." Local MongoDB service is inactive or network port 27017 is blocked. Verify that your local MongoDB daemon is running (services.msc on Windows or brew services start mongodb-community). Check connection URI string in server/.env.
CORS Blockages in Browser Console Client running on unregistered port or mismatch in CORS whitelist settings. Check the server configuration in server.js. Ensure the client's origin (e.g., http://localhost:5173) is permitted by CORS headers.
"JWT_SECRET is not defined" Missing environment variable definition in .env configuration sheet. Check that server/.env is placed directly in the server/ root directory and contains a valid JWT_SECRET string value. Restart the node server.
"Expected corresponding JSX closing tag..." Syntax errors or unclosed React JSX elements during webpack compile stage. Open the specified file in the error stack trace, locate the unclosed tag (e.g., in Projects.jsx) and correct it. Run npm run build to verify compiled assets.
